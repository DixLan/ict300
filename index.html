01 Services d'impression sous Windows Server
Le rôle Services d'impression et de numérisation de documents centralise toutes les tâches liées aux serveurs d'impression et aux imprimantes réseau. Il permet de :
  • Recevoir des documents numérisés depuis des scanneurs réseau
  • Acheminer ces documents vers une ressource réseau partagée, un site SharePoint ou des adresses e-mail
  • Gérer les files d'attente à l'impression et recevoir des notifications d'interruption
  • Migrer des serveurs d'impression et déployer des connexions via la stratégie de groupe (GPO)
🔑 Composants clés : Serveur d'impression, Serveur de télécopie, Serveur de numérisation distribuée
Le serveur de télécopie envoie et reçoit des télécopies. Il gère les ressources de télécopie (tâches, paramètres, rapports, télécopieurs) depuis un point central.

Avantage administratif : L'administrateur peut installer, gérer et contrôler les télécopieurs de plusieurs ordinateurs simultanément et à distance, en 2 ou 3 étapes — sans agir sur chaque poste client individuellement. Le Gestionnaire de service de télécopie permet d'établir automatiquement des connexions et de les mettre à disposition des utilisateurs.
Le Serveur de numérisation distribuée reçoit les documents numérisés en provenance de scanneurs réseau et les achemine vers les destinations appropriées.

Il inclut le composant logiciel enfichable Gestion de la numérisation, qui permet de gérer les scanneurs réseau et de configurer les numérisations (règles d'acheminement, destinations).
UNC = Universal Naming Convention

C'est une syntaxe standard pour désigner une ressource réseau partagée sous Windows, sans avoir à mapper un lecteur réseau.

Format : \\nom_du_serveur\nom_du_partage\chemin_optionnel
Exemple : \\SRV-PRINT\Imprimante_RDC
GPO = Group Policy Object (Objet de Stratégie de Groupe)

Une GPO est un ensemble de règles et de configurations appliquées automatiquement aux utilisateurs et aux ordinateurs d'un domaine Active Directory.

Dans le contexte des services d'impression, les GPO permettent de déployer automatiquement des connexions à des imprimantes sur les postes clients, sans intervention manuelle.
02 Service DNS sous Linux — BIND9
Le DNS (Domain Name System) est un service TCP/IP qui fait correspondre un nom de domaine qualifié (FQDN) à une adresse IP. Par exemple : www.ubuntu-fr.org → 193.55.221.76

Ubuntu est livré par défaut avec BIND (Berkeley Internet Naming Daemon), le serveur DNS le plus utilisé sur Internet, également appelé BIND9.
Serveur cache : met en cache les réponses DNS pour accélérer les résolutions suivantes et réduire la bande passante. Utile pour les connexions lentes.

Serveur maître : contient les enregistrements DNS d'un nom de domaine (zone). Référence principale pour la zone.

Serveur esclave : copie la/les zones du serveur maître. Assure la disponibilité DNS même si le maître est hors ligne. Recommandé sur les grands réseaux.

Serveurs furtifs (stealth) : identiques aux serveurs maître/esclave mais non visibles depuis Internet — uniquement internes au domaine. Le maître furtif fait les éditions de zone ; les esclaves publics distribuent les réponses.

💡 Un serveur BIND9 peut combiner plusieurs rôles simultanément (ex. cache + maître).
Récursif : le serveur interroge lui-même, tour à tour, les serveurs DNS nécessaires jusqu'à obtenir la réponse complète, puis la transmet au client.

Non-récursif (défaut BIND9) : le serveur délègue la résolution à un autre serveur DNS sans effectuer lui-même toutes les étapes.

Pour activer la récursivité, modifier /etc/bind/named.conf.options : 
allow-recursion { any; };
A (Address) : fait correspondre une adresse IP à un nom de machine. 
www      IN    A      1.2.3.4
CNAME (Alias) : crée un alias pointant vers un enregistrement A. Les chaînes de CNAME→CNAME sont déconseillées (double les requêtes). 
DNS-SRV  IN    CNAME  server
MX (Mail Exchange) : définit quel serveur reçoit les e-mails pour le domaine, avec priorité (valeur basse = priorité haute). Doit pointer vers un A, pas un CNAME. 
@    IN    MX  10  mail1.exemple.com.
@    IN    MX  20  mail2.exemple.com.
NS (Name Server) : désigne les serveurs faisant autorité pour la zone. Doit pointer vers un A. 
     IN    NS     server.ict.lan.
Les fichiers sont stockés dans /etc/bind/ :
  • named.conf — fichier principal, inclut les trois autres via include
  • named.conf.options — options globales (forwarders, DNSSEC, récursivité, écoute IPv6…)
  • named.conf.local — déclaration des zones locales (maître, esclave…)
  • named.conf.default-zones — zones par défaut (loopback, etc.)
🔑 Les fichiers de zone eux-mêmes (ex. for.ict.lan.hosts) sont référencés dans named.conf.local et stockés dans /etc/bind/.
Le SOA (Start Of Authority) est le premier enregistrement d'un fichier de zone. Il définit l'autorité de la zone et les paramètres de synchronisation maître/esclave : 
@ IN SOA server.ict.lan. root.ict.lan. (
    2018022209  ; Serial  — doit être incrémenté à chaque modification
    604800      ; Refresh — intervalle (en s) auquel l'esclave interroge le maître
    86400       ; Retry   — délai de retry si le maître est injoignable
    2419200     ; Expire  — durée après laquelle l'esclave cesse de répondre
    604800 )    ; Negative Cache TTL
Important : le numéro de série (serial) doit être incrémenté à chaque modification du fichier de zone pour que les esclaves détectent le changement.
  • DNS = Domain Name System
  • FQDN = Fully Qualified Domain Name (nom de domaine complet, ex. www.ubuntu-fr.org)
  • BIND = Berkeley Internet Naming Daemon
  • TTL = Time To Live (durée de vie d'un enregistrement en cache, en secondes)
  • SOA = Start Of Authority (enregistrement déclarant l'autorité sur une zone)
  • MX = Mail Exchange (serveur de messagerie de la zone)
  • NS = Name Server (serveur de noms faisant autorité)
  • CNAME = Canonical Name (alias vers un enregistrement A)
Le service DNS écoute sur le port 53, en UDP et TCP.

UDP port 53 : utilisé pour les requêtes/réponses standard (légères, rapides).
TCP port 53 : utilisé lorsque la réponse dépasse 512 octets (ex. transferts de zone entre serveurs maître et esclave, réponses DNSSEC volumineuses). 🔑 À retenir : port 53 UDP = requêtes DNS classiques · port 53 TCP = transferts de zone (AXFR)
La directive $TTL (Time To Live) définit la durée de vie par défaut (en secondes) des enregistrements de la zone dans le cache des serveurs DNS qui les ont résolus.

Exemple : 
$TTL    604800   ; = 7 jours (604800 secondes)
Un TTL élevé → moins de requêtes vers le serveur, mais les modifications mettent plus de temps à se propager.
Un TTL faible → propagation rapide des changements, mais plus de requêtes DNS.
Dnsmasq est un serveur léger qui fournit simultanément les services DNS, DHCP, Bootstrap Protocol et TFTP pour un petit réseau ou un poste de travail. Il est souvent actif par défaut via NetworkManager.

Pour éviter qu'il interfère avec BIND9 (conflit sur le port 53), il faut commenter la ligne suivante dans : 
/etc/NetworkManager/NetworkManager.conf
#dns=dnsmasq
(mettre un # devant dns=dnsmasq)
03 Service DHCP sous Linux — ISC-DHCP-Server
Les interfaces réseau du serveur doivent être configurées obligatoirement en adresses IP statiques avant d'installer le service DHCP. Un serveur DHCP ne peut pas s'attribuer lui-même une adresse dynamique. 
sudo apt-get install isc-dhcp-server
Le fichier de configuration principal est : /etc/dhcp/dhcpd.conf
default-lease-time : durée de bail obtenue par un client qui ne demande pas de durée spécifique. C'est la valeur standard que l'administrateur souhaite voir utiliser.

max-lease-time : durée maximale de bail que le serveur acceptera, même si le client en demande une plus longue. Le serveur ne dépassera jamais cette limite.

Exemple : default = 600s (10 min), max = 7200s (2h)
Une réservation se fait via une directive host contenant l'adresse MAC : 
subnet 192.168.1.0 netmask 255.255.255.0 {
  deny unknown-clients;
  host client1 {
    hardware ethernet DD:GH:DF:E5:F7:D7;
    fixed-address 192.168.1.20;
  }
}
deny unknown-clients : interdit l'attribution d'une adresse IP à toute station dont l'adresse MAC n'est pas déclarée dans dhcpd.conf. Renforce la sécurité.
Modifier la ligne INTERFACES dans le fichier de configuration du service (généralement /etc/default/isc-dhcp-server) : 
INTERFACES="enp0s3 enp0s8"
Pour des systèmes classiques : 
INTERFACES="eth0 eth1"
  • DHCP = Dynamic Host Configuration Protocol (protocole d'attribution automatique de configuration réseau)
  • ISC = Internet Systems Consortium (organisation qui développe et maintient ISC-DHCP et BIND)
  • MAC = Media Access Control (adresse physique unique d'une carte réseau, format XX:XX:XX:XX:XX:XX)
  • Port 67 UDP : port d'écoute du serveur DHCP (reçoit les requêtes des clients)
  • Port 68 UDP : port d'écoute du client DHCP (reçoit les réponses du serveur)
Moyen mnémotechnique : 67 = Serveur · 68 = Client (S avant C, 67 avant 68)
Deux commandes courantes : 
ifconfig | grep HWaddr      # ancienne syntaxe
ip link show                # commande moderne iproute2
L'adresse MAC apparaît sous la forme ether xx:xx:xx:xx:xx:xx.

Elle est nécessaire pour configurer des réservations d'adresses (hardware ethernet) dans dhcpd.conf.
04 Authentification Kerberos
Kerberos V5 (développé au MIT dans les années 80) repose sur :
  • Système de tickets (TGT et ST) pour authentifier les requêtes
  • Cryptographie à clés secrètes (symétrique)
  • Authentification mutuelle (le client et le serveur se vérifient mutuellement)
  • Tickets limités dans le temps (expiration automatique)
  • Mécanismes anti-rejeux (prevention du replay attack)
⚠️ Kerberos V4 et V5 sont non-interopérables.
Architecture 3-tiers :
  • Un client
  • Un serveur de ressources
  • Une autorité approuvée (AA) — reconnue de confiance par les deux autres
Principal : toute entité Kerberos identifiable (utilisateur, client, serveur) avec un nom unique.

Autorité approuvée (AA) : stocke les informations de sécurité des principaux et génère/gère les clés de session.

Royaume : organisation logique dans laquelle s'exécute au moins une AA, capable d'authentifier tous les principaux déclarés.

KDC (Key Distribution Center) : nom donné à l'AA dans l'environnement Windows 2000/Active Directory.
Un ticket est une structure de données composée d'une partie chiffrée et d'une partie en clair.

TGT (Ticket Granting Ticket) :
  • Obtenu lors de l'authentification initiale auprès du service AS
  • Permet d'obtenir des ST sans ressaisir le mot de passe
  • Chiffré avec la clé secrète du client (seul le bon client peut le déchiffrer)
ST (Service Ticket) :
  • Obtenu en présentant le TGT au service TGS
  • Permet d'accéder à une ressource spécifique
  • Valide pour un seul service
Étape 1 — Le client envoie une requête d'authentification (en clair : identité + serveur cible) au service AS.
Étape 2 — L'AS émet un TGT, chiffré avec la clé secrète (Ksec) du client.
Étape 3 — Le client présente le TGT au service TGS pour demander un ST.
Étape 4 — Le TGS émet un ST pour le service demandé.
Étape 5 — Le client présente le ST au serveur de ressources.
Étape 6 — Le serveur de ressources valide le ST et accorde (ou refuse) l'accès.

🔑 Les clés secrètes ne servent qu'à l'étape initiale. Les autres phases utilisent des clés de session jetables.
AS (Authentication Service) : gère l'authentification initiale du client et émet le TGT.

TGS (Ticket Granting Service) : reçoit les TGT valides et émet les ST pour les services demandés.

Ces deux services ne tournent pas nécessairement sur le même serveur — ils peuvent être distribués sur des machines distinctes pour raisons de disponibilité et de sécurité.
  • KDC = Key Distribution Center (nom Windows de l'autorité approuvée)
  • TGT = Ticket Granting Ticket (ticket d'authentification initial)
  • ST = Service Ticket (ticket d'accès à un service spécifique)
  • AS = Authentication Service (service d'authentification initiale)
  • TGS = Ticket Granting Service (service d'octroi de tickets de service)
  • AA = Autorité Approuvée (serveur de confiance central)
  • Ksec = Clé secrète du principal (dans Windows, dérivée directement du mot de passe utilisateur)
RFCs : Kerberos V5 est standardisé par les RFC 1510 et 1964 de l'IETF.

Origine : développé au MIT (Massachusetts Institute of Technology) dans le cadre du projet Athena, au début des années 1980.

💡 Le projet Athena visait à fournir une infrastructure informatique décentralisée et sécurisée à l'université.
Lorsque le client reçoit un ST, il doit préparer une requête pour le serveur de ressources. Il crée un authentifiant : une structure de données chiffrée avec la clé de session contenue dans le ticket.

Le client envoie au serveur de ressources :
  • Le ST (chiffré avec la clé du serveur de ressources)
  • L'authentifiant (chiffré avec la clé de session)
Le serveur déchiffre le ST pour obtenir la clé de session, puis vérifie l'authentifiant. Si valide → accès accordé ; sinon → accès refusé.

Ce mécanisme évite les attaques par rejeu car l'authentifiant contient un timestamp.
05 Introduction au système de fichiers NFS
NFS (Network File System) est un protocole de système de fichiers réseau permettant à un ordinateur d'accéder via un réseau à des fichiers distants.

  • Origine : développé par Sun Microsystems en 1984
  • Couche OSI : couche Application
  • Protocole utilisé : RPC (Remote Procedure Call)
  • Cible principale : systèmes UNIX/Linux (des versions existent pour Macintosh et Windows)
  • Compatible avec IPv6 sur la plupart des systèmes
Améliorations de NFSv4 :
  • Gestion totale de la sécurité (négociation, support Kerberos 5, SPKM, LIPKEY4, chiffrement possible)
  • Support accru de la montée en charge (groupement de requêtes compound, délégation client)
  • Migration et réplication transparentes
  • Reprise sur incident intégrée côté client et serveur
  • Compatibilité Unix et Windows (Mac depuis OS X Lion 10.7)
  • Support de plusieurs protocoles de transport (TCP, RDMA)
Incompatibilités :
  • NFSv4 est statefull (à état) → incompatible avec NFSv3 (stateless)
  • NFSv4 ne supporte pas UDP
  • Suppose l'utilisation d'UTF-8 pour les noms de fichiers
✅ Il est hautement préférable d'utiliser NFSv4 si une migration totale est possible.
NFSv4.1 (RFC 5661, janvier 2010) s'inspire de pNFS, Lustre et HTTP. Il introduit :
  • Sessions : interruptibles et rétablissables, simplifient le failover
  • Délégation par répertoires
  • Abstraction du transport : indépendant de IP et de TCP
  • Parallélisation des accès aux fichiers (Striping)
  • Géométrie et segments de fichiers abstraits, chemins multiples vers les données
  • Optimisation des fichiers creux
  • NFS = Network File System (système de fichiers réseau)
  • RPC = Remote Procedure Call (protocole d'appel de procédures à distance, utilisé par NFS pour ses communications)
  • pNFS = parallel NFS (extension permettant l'accès parallèle aux données, inspiration de NFSv4.1)
  • RDMA = Remote Direct Memory Access (protocole de transport haute performance supporté par NFSv4, sans passer par le CPU)
  • UTF-8 = Unicode Transformation Format 8-bit (encodage de caractères supposé par NFSv4 pour les noms de fichiers)
NFSv4.1 est défini dans le RFC 5661, publié en janvier 2010.

Il s'est inspiré de :
  • pNFS (parallel NFS)
  • Lustre (système de fichiers distribué haute performance)
  • Protocoles internet comme HTTP
💡 NFSv4.2 était en développement et de nouvelles révisions sont régulièrement proposées.
Statefull (à état) : le serveur conserve des informations sur les sessions actives et les opérations en cours entre les requêtes.

NFSv4 = statefull : maintient l'état des connexions → permet la reprise sur incident (failover) et la délégation de fichiers au client.

NFSv3 = stateless (sans état) : chaque requête est indépendante, le serveur ne retient aucun contexte entre deux appels → plus simple mais moins robuste en cas de panne.

Cette différence fondamentale rend NFSv4 incompatible avec NFSv3.
06 Installation et configuration NFS — Windows
Prérequis :
  • Installer Active Directory
  • Installer le service NFS Serveur (via le Gestionnaire de serveur)
  • Optionnellement installer la partie Client NFS
Configuration dans les paramètres généraux :
Dans "Services pour NFS" → clic droit → Paramètres généraux → cocher le Nom de domaine Active Directory (ex. ict.lan).

Dans l'onglet Netgroups, il est possible d'activer les netgroups NIS ou le domaine Active Directory.
Sur Windows 10 :
  • Installer le service Client pour NFS (via Fonctionnalités Windows)
  • Connecter le partage NFS en utilisant une connexion standard (UNC) comme pour un dossier partagé classique
Microsoft supporte NFS nativement, et l'accès peut se faire de façon transparente depuis l'explorateur de fichiers Windows, sans logiciel tiers.
NIS = Network Information Service (anciennement Yellow Pages — YP), un service réseau Unix permettant de centraliser la gestion des comptes utilisateurs, mots de passe et configuration.

Un netgroup est un groupe de machines ou d'utilisateurs défini dans NIS. Dans le fichier /etc/exports de NFS, on peut restreindre l'accès à un partage à un netgroup entier avec la notation @nom_netgroup, plutôt que de lister chaque IP individuellement.

Dans l'interface Windows Server NFS, l'onglet Netgroups permet d'activer soit les netgroups NIS, soit le domaine Active Directory.
07 Installation et configuration NFS — Linux
Paquet serveur : 
sudo apt-get install nfs-kernel-server
Fichier de configuration des partages : /etc/exports

Exemple de syntaxe : 
/dossier/partage  192.168.0.0/24(rw,all_squash,anonuid=1000,anongid=1000,sync,no_subtree_check)
Après modification, recharger le service : 
sudo service nfs-kernel-server reload
Vérifier les exports actifs : 
showmount -e
rw : lecture et écriture (par défaut : ro = lecture seule)
sync : le serveur respecte le protocole NFS — répond après que les données soient écrites sur le disque. Sécurisé.
async : répond avant l'écriture réelle → meilleures performances, mais risque de corruption de données en cas de crash.
root_squash : mappe l'utilisateur root côté client vers l'utilisateur anonyme (option par défaut — sécurité).
no_root_squash : l'utilisateur root du client garde ses privilèges root sur le serveur (dangereux).
all_squash : tous les utilisateurs sont mappés vers l'utilisateur anonyme.
anonuid / anongid : définit l'UID et le GID de l'utilisateur anonyme utilisé par les options de mapping.
no_subtree_check : désactive la vérification de sous-répertoires (améliore fiabilité, légère implication sécurité).
1. Installer le paquet client : 
sudo apt-get install nfs-common
2. Créer le point de montage : 
sudo mkdir /media/NFS
3. Ajouter dans /etc/fstab : 
192.168.xx.10:/dossier_partage/   /media/NFS   nfs   defaults,user,auto,noatime,intr   0 0
  • 192.168.xx.10 = IP du serveur NFS
  • /media/NFS = point de montage local
  • nfs = protocole
  • 0 0 = pas de vérification d'erreurs disque au démarrage
Explorer les partages disponibles sur le serveur : 
showmount -e serveur.local
sudo service nfs-kernel-server reload   # relit /etc/exports sans couper le service
sudo service nfs-kernel-server restart  # redémarre complètement le service
reload : préférable — prend en compte les modifications du fichier /etc/exports sans interrompre les transferts en cours ni déconnecter les clients montés.

restart : nécessaire si l'export n'est pas actif après un reload. Attention : interrompt les transferts en cours et déconnecte les clients NFS montés.
noatime : ne met pas à jour l'horodatage d'accès (access time) à chaque lecture de fichier → réduit les écritures inutiles et améliore les performances.

intr : permet d'interrompre (avec Ctrl+C) une opération NFS bloquée, utile si le serveur devient inaccessible.

Les deux 0 0 finaux :
  • 1er 0 = pas de sauvegarde par dump
  • 2e 0 = pas de vérification d'erreurs disque par fsck au démarrage (approprié pour les montages réseau)
  • UID = User Identifier — identifiant numérique unique d'un utilisateur Linux (ex. root = 0, premier utilisateur créé = 1000)
  • GID = Group Identifier — identifiant numérique unique d'un groupe Linux
Dans /etc/exports, anonuid=1000 et anongid=1000 indiquent au serveur NFS que l'utilisateur anonyme (résultat des options all_squash ou root_squash) doit être traité avec l'UID et le GID 1000, ce qui correspond généralement au premier utilisateur non-root du système. 
sudo id                  # pour connaître son UID/GID
08 Introduction au système de fichiers SMB / Samba
Samba est un logiciel libre qui implémente le protocole propriétaire SMB/CIFS de Microsoft Windows sur des systèmes Unix/Linux. Il permet de partager des fichiers et des imprimantes dans un réseau hétérogène Windows-Unix.

  • Développé initialement par l'australien Andrew Tridgell
  • Distribué sous licence GNU GPL
  • Son nom vient de SMB (Server Message Block) avec deux voyelles ajoutées → SaMBa
  • Fonctionne sur Linux, Solaris, AIX, BSD, Mac OS X
  • Permet aux serveurs Unix de se substituer à des serveurs Windows
Samba implémente une dizaine de services et protocoles :
  • NetBIOS sur TCP/IP (NBT)
  • SMB / CIFS (système de fichiers réseau)
  • DCE/RPC / MSRPC
  • WINS (serveur de noms NetBIOS)
  • SAM (Security Account Manager)
  • LSA (Local Security Authority)
  • Spoolss (service d'impression)
  • NTLM et Kerberos (authentification)
  • LDAP (annuaire, dans Samba 4)
Samba 4 inclut un KDC Kerberos et un serveur LDAP complets.
Lorsque NFS et Samba sont installés en parallèle, Samba est moins performant que NFS en termes de taux de transfert.

Cependant, une étude a montré que Samba 3 est jusqu'à 2,5 fois plus rapide que l'implémentation SMB de Windows Server 2003.

Samba 3.2 apporte :
  • Support de IPv6
  • Possibilité de stocker les partages dans la base de registre
  • Expérimentation du clustering
La configuration est centralisée dans un fichier unique : /etc/samba/smb.conf
  • SMB = Server Message Block (protocole propriétaire Microsoft de partage de fichiers et d'imprimantes)
  • CIFS = Common Internet File System (extension de SMB, système de fichiers via SMB)
  • SWAT = Samba Web Administration Tool (outil d'administration web inclus dans Samba)
  • WINS = Windows Internet Name Service (serveur de noms NetBIOS, résolution de noms NetBIOS → IP)
  • NBNS = NetBIOS Name Server (autre nom pour WINS)
  • NBT = NetBIOS over TCP/IP (encapsulation NetBIOS dans TCP/IP)
  • SAM = Security Account Manager (base de données locale des comptes et mots de passe Windows)
  • LSA = Local Security Authority (composant Windows gérant les politiques de sécurité locales)
Le partage netlogon (chemin par défaut : /etc/samba/netlogon) est le répertoire de scripts d'ouverture de session utilisateur.

Il est généralement accessible en lecture seule. Lorsqu'un utilisateur Windows se connecte au domaine, son poste contacte ce partage pour y récupérer et exécuter un script d'ouverture de session (ex. mappage de lecteurs réseau, configuration d'imprimantes).
  • DCE/RPC = Distributed Computing Environment / Remote Procedure Call — framework standard de communication entre processus distribués sur un réseau.
  • MSRPC = Microsoft Remote Procedure Call — implémentation propriétaire Microsoft de DCE/RPC, utilisée par Windows pour ses services réseau (Active Directory, services d'impression, SAM…).
Samba implémente MSRPC pour être capable d'interagir avec les systèmes Windows comme un vrai serveur Windows.
09 Installation et configuration de Samba
3 modes possibles :
  • Workgroup (groupe de travail) : serveur autonome, sans domaine
  • Serveur membre de domaine : intégré à un domaine existant (Active Directory ou NT)
  • Contrôleur de domaine : gère l'authentification du domaine
Limitation importante : il n'est pas possible de transformer un serveur Samba en contrôleur de domaine dans un domaine Microsoft (Active Directory natif Windows). 
sudo apt-get update
sudo apt-get install samba
Fichier de configuration : /etc/samba/smb.conf

Bonne pratique — sauvegarder avant de modifier : 
sudo cp -a /etc/samba/smb.conf /etc/samba/smb_backup.conf
Pour éditer : sudo nano /etc/samba/smb.conf
Dans la section [global] : 
workgroup = EUREKA
security = user

Puis ajouter une section de partage : 
[share]
    comment = Dossier partagé sur le serveur Ubuntu
    path = /srvubuntu/samba/share
    browsable = yes
    guest ok = yes
    read only = no
    create mask = 0755
Redémarrer le service après modification : 
sudo service smbd restart
Depuis Ubuntu (Nautilus) :
Fichiers → Connexion à un serveur → adresse : 
smb://srvubuntu/share
Depuis Windows :
Explorateur de fichiers → Connecter un lecteur réseau → entrer le chemin UNC : 
\\srvubuntu\share
ou directement dans la barre d'adresse de l'explorateur. 💡 Les droits du dossier côté Linux doivent correspondre au create mask défini dans smb.conf (ex. 0755). 
sudo service smbd start    # Démarrer
sudo service smbd stop     # Arrêter
sudo service smbd restart  # Redémarrer (interrompt les connexions actives)
⚠️ Le daemon Samba s'appelle smbd. Ne pas confondre avec nmbd (service NetBIOS).
create mask définit les permissions UNIX appliquées aux fichiers créés via le partage Samba.

En notation octale Linux : 
0 7 5 5
  │ │ │
  │ │ └─ Autres (others) : lecture + exécution (r-x)
  │ └─── Groupe (group)  : lecture + exécution (r-x)
  └───── Propriétaire    : lecture + écriture + exécution (rwx)
  • 7 = rwx = 4+2+1 (le propriétaire peut tout faire)
  • 5 = r-x = 4+0+1 (groupe et autres peuvent lire et exécuter)
Pour un partage en lecture/écriture pour tous : create mask = 0777
  • browsable = yes : le partage est visible dans le voisinage réseau Windows. Si no, il est caché mais toujours accessible via son chemin exact.
  • guest ok = yes : autorise les connexions anonymes (sans identifiant ni mot de passe). Si no, une authentification est requise.
  • read only = no : accès en lecture et écriture. Si yes (valeur par défaut), le partage est en lecture seule.
security = user : chaque utilisateur doit s'authentifier avec un nom d'utilisateur et un mot de passe Samba avant d'accéder aux partages. Mode le plus courant pour un serveur autonome.

Les comptes Samba sont indépendants des comptes Linux (gérés avec smbpasswd).

Ancien mode : security = share (déprécié, supprimé dans Samba 4) — l'accès est basé sur le partage lui-même, sans identification utilisateur. Moins sécurisé. Dans l'exemple du cours, guest ok = yes combiné avec security = user permet un accès invité sans mot de passe pour ce partage précis.
